Detalles Técnicos de Cifrado

1. Arquitectura Criptográfica

STC Cloud implementa una arquitectura de seguridad basada en el principio de Defensa en Profundidad. Los secretos del agente (tokens de API, credenciales de proxy) nunca se almacenan en texto plano.

Cifrado Simétrico

  • Algoritmo: AES-256-GCM (Advanced Encryption Standard en modo Galois/Counter Mode).
  • Autenticación: El modo GCM garantiza tanto la confidencialidad como la integridad del payload cifrado mediante un tag de autenticación de 128 bits.
  • Vectores de Inicialización (IV): Se generan 12 bytes aleatorios por cada operación de cifrado para prevenir ataques de diccionario.

Derivación de Clave (KDF)

  • Algoritmo: PBKDF2-SHA256 (Password-Based Key Derivation Function 2).
  • Iteraciones: 100,000 iteraciones para dificultar ataques de fuerza bruta.
  • Salt: Se utiliza un salt único derivado de la identidad del host.

2. Hardware Binding (Anti-Clonación)

Para prevenir que un agente sea movido o clonado entre servidores, la clave de cifrado se vincula físicamente al hardware del host mediante la concatenación de:

  1. Windows MachineGuid: Identificador único de la instalación del SO.
  2. BIOS Serial Number: Identificador físico de la placa base/firmware.

Si se intenta mover el archivo de configuración a una máquina diferente, el proceso de derivación resultará en una clave distinta y el descifrado de los secretos fallará, invalidando el acceso del agente.

Seguridad de Tráfico

3. Seguridad en Comunicaciones

Transporte Seguro

Toda la comunicación externa (API REST y WebSockets) se realiza forzosamente sobre TLS 1.2 o superior utilizando suites de cifrado modernas con Forward Secrecy.

Autenticación Basada en Tokens

  • Mecanismo: JSON Web Tokens (JWT).
  • Firma: Algoritmo asimétrico RS256 (RSA con SHA-256).
  • Vigencia: Los tokens tienen una validez de 30 días, tras los cuales el agente debe re-validar su Hardware ID con el servidor.

Aislamiento de Red

El agente solo requiere visibilidad de red hacia las impresoras (SNMP) y salida HTTPS a la nube. No escucha en ninguna interfaz de red externa (Inbound ports: 0).

Integridad de Binarios

4. Integridad de Actualizaciones (Ed25519)

El sistema de auto-update implementa una cadena de confianza para prevenir ataques de tipo Supply Chain.

Firma Digital de Paquetes

Cada paquete de actualización (`bundle.js`) se distribuye junto a una firma criptográfica externa. El agente no aplicará ninguna actualización que no pase la validación:

  • Algoritmo de Firma: Ed25519 (Criptografía de Curva Elíptica).
  • Proceso de Verificación: El agente posee la llave pública SPKI (`UPDATE_PUBLIC_KEY_HEX`) integrada en su binario original. Al descargar una actualización, se descarga su archivo `.sig` correspondiente y se verifica la integridad del buffer antes de su escritura en disco.
  • Hash de Integridad: Adicionalmente, se realiza una validación de hash SHA-256 proporcionada por la API central.
Protección ante Tampering: Si un atacante intercepta la conexión o el almacenamiento de actualizaciones y modifica un solo byte del paquete, la firma Ed25519 será inválida y el agente rechazará la instalación automáticamente.

Privacidad de Datos

5. Protección de Datos en Reposo

Base de Datos Local

El agente utiliza SQLite para la cola de mensajes offline. Importante: Solo se almacenan datos técnicos de dispositivos (lecturas de contadores, niveles de suministros). No se almacena información sensible de usuarios finales ni nombres de documentos impresos.

Privacidad (GDPR/Compliance)

El agente está diseñado bajo el principio de Privacidad desde el Diseño. La telemetría recolectada es puramente técnica y no permite la identificación de personas físicas, cumpliendo con los estándares internacionales de protección de datos.
