Requisitos del Sistema

Introducción

STC Cloud es una solución de gestión de flotas de impresión de nivel empresarial. La arquitectura se basa en el despliegue de micro-agentes (STC Cloud Monitor) que interactúan con los dispositivos mediante protocolos estándar de la industria, garantizando una recolección de datos precisa y en tiempo real.

La solución consta de dos componentes: el STC Cloud Portal (interfaz web de gestión centralizada) y el STC Cloud Monitor (agente ligero instalado en cada site). El agente opera de forma autónoma, con resiliencia ante pérdida de conectividad, soporte de proxy HTTP corporativo, y se actualiza automáticamente desde el servidor central sin intervención del administrador.

Sistemas operativos

  • Windows Server 2016 / 2019 / 2022 (Todas las ediciones)
  • Windows 10 / 11 Pro / Enterprise (64-bit)

Nota: El agente se instala y ejecuta como servicio Windows bajo la cuenta "LocalSystem". Compatible con entornos virtualizados (VMware, Hyper-V). Se admite la instalación en hosts con múltiples interfaces de red (multi-NIC); Windows determina qué NIC utiliza el agente para el tráfico saliente.

Hardware

El Agente STC Cloud es un servicio ligero diseñado para ejecutarse con un impacto mínimo en los recursos.

Componente Requisito Mínimo Configuración Recomendada
Procesador Dual Core 2.0 GHz Quad Core (Intel i3 / Xeon)
Memoria RAM 2 GB 4 GB
Almacenamiento 200 MB Libre 1 GB (Para persistencia de logs)

Configuración de Seguridad

Preparando el entorno

Antes de la instalación del Agente STC Cloud, asegúrese de que el entorno cumpla con las siguientes políticas de seguridad y acceso:

Privilegios de Ejecución

  • El agente debe instalarse con privilegios de Administrador Local.
  • El servicio se ejecuta bajo la cuenta "LocalSystem" para garantizar persistencia y acceso al hardware.

Seguridad de Datos (Cifrado)

STC Cloud utiliza un sistema de cifrado de grado militar para proteger la identidad del agente y su configuración:

  • AES-256-GCM: Utilizado para el almacenamiento de tokens de acceso y configuración de red local. La clave se deriva mediante PBKDF2-SHA256 con 100.000 iteraciones.
  • Hardware Binding: La clave de cifrado se deriva de atributos físicos del servidor (MachineGuid y BIOS Serial), impidiendo la clonación del agente entre máquinas.

Directorio de Trabajo

El agente requiere permisos de lectura/escritura en el siguiente directorio:

C:\ProgramData\STCCloudMonitor\

Soporte de Proxy HTTP Corporativo

En entornos donde el acceso a Internet está restringido por un servidor proxy corporativo, el agente puede configurarse para enrutar todas sus comunicaciones (HTTPS y WebSockets) a través de dicho proxy.

  • Configuración: vía la pestaña "Network / Proxy" en la consola STC Cloud Monitor Console, o mediante el flag CLI --set-proxy.
  • Formato de URL: http://[usuario:contraseña@]host:puerto
  • Autenticación: Basic Authentication (RFC 7235) — las credenciales se almacenan cifradas con AES-256-GCM junto al resto de la configuración del agente.
  • Protocolo de túnel: el agente establece un túnel CONNECT hacia el servidor proxy para encapsular tanto el tráfico HTTPS como las conexiones WebSocket (WSS). Implementado sin dependencias externas adicionales.
Nota para administradores: Si se usan credenciales de proxy, asegúrese de que las políticas de seguridad (restablecimientos de contraseña, expiración de sesión) no interrumpan la conexión del servicio. Se recomienda usar una cuenta de servicio dedicada con contraseña sin expiración.

Apéndice 1: Puertos

Apéndice 1: información detallada del puerto para STC Monitor

El monitor STC requiere las siguientes aperturas de puerto para una funcionalidad completa:

Puerto Protocolo Dirección Descripción
443 TCP (HTTPS) Saliente — Internet Comunicación segura con la API Cloud: telemetría, heartbeat y activación.
443 TCP (WSS) Saliente — Internet Canal de control en tiempo real (WebSockets sobre TLS). Reconexión con Exponential Backoff.
161 UDP (SNMP) Saliente — LAN Descubrimiento y lectura de contadores de impresoras (SNMP v2c).
9100 / 80 / 443 TCP Saliente — LAN Pre-verificación de alcanzabilidad TCP antes del escaneo SNMP. El agente prueba estos tres puertos en paralelo (JetDirect, HTTP EWS, HTTPS EWS) con timeout de 500 ms. Si al menos uno responde, se procede con SNMP. No se requieren los tres; basta con que uno sea accesible.
8000 TCP Localhost Bridge para diagnóstico de terminal remota (Consola). Solo escucha en 127.0.0.1.
Arquitectura Zero-Inbound: No se requieren puertos de entrada (Inbound) desde Internet. El agente inicia todas las conexiones hacia la nube. El puerto 8000 solo acepta conexiones locales (loopback).

Los puertos 9100 / 80 / 443 hacia la LAN son utilizados únicamente para la pre-verificación de alcanzabilidad. Esta verificación reduce el tiempo de escaneo al descartar IPs vacías (timeout de 500 ms vs. 3 segundos de SNMP). Si el firewall interno bloquea los tres puertos hacia las impresoras, el agente omitirá la pre-verificación e intentará SNMP directamente.

Información de URL

Información de la URL del monitor STC

El agente debe tener acceso a los siguientes dominios para operar correctamente. Se recomienda añadirlos a la lista de permitidos (Allowlist) de su Firewall/Proxy:

Endpoints Principales

URL / Dominio Función Frecuencia
stc-cloud-portal.vercel.app Portal de Gestión (Acceso Web) Bajo demanda
stc-cloud-api.onrender.com API de Datos, Telemetría y Control Cada 60 s (Heartbeat)
stc-cloud-api.onrender.com
/api/v1/agents/version		 Consulta de versión para auto-update Al iniciar + cada 4 h

Validación de Conectividad al Inicio

Al iniciar el servicio, el agente realiza una prueba de conectividad HTTPS contra el endpoint /api/v1/health antes de arrancar los loops de monitoreo. Si el servidor no es alcanzable, el agente entra en modo de reintento automático con Exponential Backoff: primer reintento a los 10 s, duplicando el intervalo en cada intento hasta un máximo de 5 minutos. Este comportamiento previene la sobrecarga del servidor y el consumo innecesario de recursos durante interrupciones de red o configuración incorrecta del proxy.

Actualización Automática del Agente

El agente consulta el endpoint GET /api/v1/agents/version al arrancar y cada 4 horas. Si el servidor reporta una versión estrictamente mayor (comparación semver) a la versión instalada, el agente descarga el nuevo bundle de forma transparente, lo aplica de forma atómica (escritura en archivo temporal seguida de renombrado) y se reinicia. El gestor de servicios (NSSM) reinicia el proceso automáticamente con el nuevo binario, sin intervención del administrador. El agente nunca se degrada a una versión anterior.

El administrador del servidor debe configurar las siguientes variables de entorno en la plataforma de hosting para gestionar las distribuciones:

Variable de entorno Descripción Ejemplo
AGENT_VERSION Versión actual del agente publicada 1.0.0
AGENT_DOWNLOAD_URL URL pública del bundle.js a descargar https://github.com/.../releases/download/v1.0.0/bundle.js

Ambas variables se actualizan automáticamente al ejecutar el script de build (build-installer.bat), que compila el agente, genera el instalador Windows, publica el release en GitHub y actualiza Render en un solo paso.

Cloud Signalling and On-Going Communication

STC Cloud utiliza tecnología de WebSockets persistentes (WSS sobre TLS 1.2+) para permitir la gestión remota sin latencia. El canal implementa reconexión automática con Exponential Backoff: primer reintento a los 5 s, duplicando el intervalo en cada intento hasta un máximo de 300 s (5 minutos). Al restablecer la conexión, el intervalo se resetea. Esto permite realizar acciones como:

  • Consulta de estado de agente en tiempo real.
  • Escaneo de red forzado bajo demanda.
  • Actualización remota de parámetros de configuración.
  • Reinicio del servicio del agente desde el portal.
Servicio Tecnología Seguridad Resiliencia
Ingesta de Datos REST API / JSON TLS 1.2+ Cola SQLite offline + reintentos
Comando y Control WebSockets (WSS) TLS 1.2+ / JWT Auth Exponential Backoff (5 s → 300 s)
Auto-Update HTTPS / JSON + descarga binaria TLS 1.2+ Escritura atómica · solo versiones mayores

Comportamiento Offline

Cuando el agente pierde conectividad con la nube, las lecturas SNMP continúan almacenándose localmente en una base de datos SQLite (C:\ProgramData\STCCloudMonitor\queue.db). Al restablecer la conexión, los datos acumulados se sincronizan automáticamente en lotes de hasta 500 lecturas por ciclo, garantizando que no se pierda ninguna medición.

Apéndice 3: Certificaciones

Apéndice 3 – Acreditaciones y Certificaciones

STC Cloud ha sido desarrollado bajo estrictos estándares de calidad y seguridad de la información, asegurando el cumplimiento con las normativas corporativas vigentes.

Privacidad y Cumplimiento

  • GDPR / LGPD: No se recolectan datos personales de los usuarios. Solo se procesa telemetría técnica de hardware (contadores, niveles de tóner, alertas de estado).
  • Seguridad de Red: Arquitectura Zero-Inbound (No requiere apertura de puertos de entrada desde Internet).
  • ISO 27001 Ready: Procesos de desarrollo alineados con las mejores prácticas de gestión de seguridad de la información.
  • Cifrado en Reposo: Toda la configuración sensible del agente (tokens, URL del servidor, credenciales de proxy) se almacena cifrada con AES-256-GCM + Hardware Binding. Nunca se almacena información sensible en texto plano.

Resumen de Características de Seguridad

Característica Implementación
Cifrado de config AES-256-GCM · PBKDF2-SHA256 (100k iter) · Hardware Binding
Autenticación cloud JWT RS256 · 30 días para agentes · Refresh Tokens (SHA-256)
Transporte TLS 1.2+ en todas las conexiones salientes
Proxy corporativo HTTP CONNECT tunnel · Basic Auth cifrada en reposo
Auto-update Descarga HTTPS · escritura atómica · solo versiones mayores (semver)
